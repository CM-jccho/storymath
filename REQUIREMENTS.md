# StoryMath 요구사항 (Deep Interview 결과)

작성일: 2026-07-02

## 문제 정의

아이들이 수학을 못 푸는 이유는 계산을 못해서가 아니라, **"푼다는 것"이 무엇인지(정의)와 지금 다음에 무엇을 해야 하는지(절차)를 모르기 때문**이다. 이 사이트는 문제 풀이를 "읽고 외우는 것"이 아니라 "단계를 스스로 밟아가는 것"으로 경험하게 하여 로지컬 씽킹을 기른다.

## 확정된 결정사항

### 1. 대상 범위 (사용자 확정)
- 한국 2022 개정 교육과정 기준 **초등 1학년 ~ 중등 3학년** 전체
- 학년 × 단원 × 난이도(여러 단계) 체계

### 2. 핵심 학습 경험 (사용자 확정)
- **인터랙티브 단계별 풀이**: 학습자가 각 단계를 직접 선택/입력하며 진행
  1. 이 문제에서 **알고 있는 것**은? (선택)
  2. **구해야 하는 것**은? (선택/입력)
  3. 그럼 **다음에 뭘 해야 할까**? (전략 선택)
  4. 실행 → 확인
- 막히면 단계별 힌트 제공 (정답 공개 전에 점진적 힌트)
- 문제는 스토리(서사) 기반 — "스토리 수학" 정체성

### 3. 콘텐츠 생산 (사용자 확정)
- **AI(Claude) 사전 대량 생성 + 검수 → 정적 데이터로 서빙**
- 런타임 API 호출 없음 → 운영 비용 최소화, 품질 통제 가능
- 교육과정 표준(성취기준)을 생성 프롬프트의 근거로 사용
- 추후 실시간 AI 튜터를 얹을 수 있는 구조로 설계

### 4. MVP 범위 (사용자 확정, 2026-07-02)
- 구조(교육과정 체계, 데이터 스키마, 생성 파이프라인, 사이트)는 초1~중3 전체를 담게 설계
- 콘텐츠는 **파일럿 학년 초4를 전 단원 채워 완성** → 실제 학습자 검증 → 파이프라인으로 확장

### 5. 계정/기록 (사용자 확정, 2026-07-02)
- 회원가입 없음, **브라우저 로컬 저장**(localStorage)으로 진도·오답 기록
- 서버/DB 없는 정적 사이트로 운영 → 배포·운영 부담 최소화

### 6. 풀이 방법론 (설계 결정)
- Polya 4단계 기반 표준 프레임워크: **이해(아는 것/구할 것) → 계획(전략 선택) → 실행(계산) → 반성(검산·의미 확인)**
- 모든 문제의 단계(step) 구조가 이 프레임워크에 매핑됨

### 7. 기술 스택 (설계 결정)
- 빌드 도구 없는 **순수 정적 사이트** (HTML/CSS/vanilla JS + JSON 데이터)
- 이유: iCloud 폴더에서 node_modules 없이 운영, GitHub Pages 등 어디든 배포 가능, 의존성 0
- 로컬 실행: `python3 -m http.server`

## 열린 질문

1. **"스토리 수학"**: 기존 브랜드/교재를 지칭하는지, 서사형 문제라는 컨셉인지 (현재는 컨셉으로 가정)
2. **배포처**: GitHub Pages / Vercel 등 (정적 사이트라 어디든 가능, 미정)
3. **검수 방식**: AI 생성 콘텐츠를 누가/어떻게 검수할지 (본인 직접 / AI 교차검증 + 샘플링)
